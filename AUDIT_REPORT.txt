OpenCode JCE - Full Audit Report
Date: 2026-05-04
Scope: install, update, config merge, MCP/context workflow, CLI runtime, security-sensitive logic.

Resolution status:
- 2026-05-04: All findings below have corresponding fixes in the working tree.
- Regression coverage added in tests/unit/audit-fixes.test.ts for the highest-risk runtime issues.

Verification run:
- bun run typecheck: passed, no TypeScript errors printed.
- bun test: passed, 153 pass, 0 fail, 1189 expect() calls, 7 files.
- git status --short before report: .opencode-context.md modified by required session context update.

Summary:
- The test suite passes, but multiple workflow and runtime bugs remain because they are not covered by tests.
- Highest-risk areas are self-update/install rollback safety, OpenCode-native MCP config, plugin URL validation, and fallback provider health checks.
- Many issues come from duplicated config literals across install.ps1, install.sh, scripts/merge-config.ts, and src/lib/opencode-json-template.ts.

Findings:

1. HIGH - Self-update breaks Unix/macOS CLI shim
Files:
- src/commands/update.ts:212-233
- src/commands/update.ts:630-633
Evidence:
- ensureCliShim() always targets ~/.bun/bin, removes opencode-jce, opencode-jce.exe, and opencode-jce.bunx, then writes only opencode-jce.cmd.
- selfUpdateCli() calls ensureCliShim() on every platform.
Impact:
- On Linux/macOS, update can delete the executable opencode-jce and leave only a Windows .cmd shim. The command can become unavailable after update.
Recommendation:
- Make ensureCliShim() platform-aware. On Windows, write opencode-jce.cmd. On Unix/macOS, write an executable opencode-jce shell shim and chmod 755.

2. HIGH - Update can report success after CLI self-update failure
Files:
- src/commands/update.ts:115-131
- src/commands/update.ts:630-633
- src/commands/update.ts:690-704
Evidence:
- selfUpdateCli() catches errors and returns false.
- The command action awaits selfUpdateCli(latestVersion) but ignores the boolean result.
- The update continues to merge config and prints Update Complete.
Impact:
- A failed CLI update can still be marked complete, and version.json can be advanced to latestVersion. Users may believe they are updated while running stale or partially broken CLI code.
Recommendation:
- Treat false from selfUpdateCli() as fatal unless an explicit --config-only mode exists. Do not update version.json when the CLI update failed.

3. HIGH - Install/update can remove the working CLI before replacement is proven valid
Files:
- src/commands/update.ts:171-184
- install.ps1:449-458
- install.sh:332-340
Evidence:
- updateLocalCliFolder() removes old src, schemas, and scripts before all replacement files and dependencies are verified.
- install.ps1 removes the whole cli directory before copying the replacement.
- install.sh rm -rf's the whole cli directory before copying the replacement.
Impact:
- Interrupted copy, missing temp files, permission failures, or dependency failures can leave users without a working opencode-jce CLI.
Recommendation:
- Stage into a temp directory such as cli.new, verify required files and dependency install, then atomically swap with rollback to the previous cli directory.

4. MEDIUM - Dependency install failure is ignored during update
File:
- src/commands/update.ts:194-204
Evidence:
- Bun.spawn(["bun", "install"]) is awaited, but its exit code is not checked.
- success("CLI source updated.") is printed afterward regardless of install result.
Impact:
- Update can succeed with missing or stale dependencies.
Recommendation:
- Capture const installExit = await installProc.exited and throw if installExit !== 0.

5. MEDIUM - Newer local versions are treated as current and can be downgraded
File:
- src/commands/update.ts:611-617
- src/commands/update.ts:630-633
- src/commands/update.ts:690-699
Evidence:
- compareVersions(latestVersion, localVersion) is used.
- Any comparison <= 0 prints "Version is current. Syncing latest files..." and still clones main and updates version.json to latestVersion.
Impact:
- A local dev/prerelease build newer than GitHub main can be overwritten and marked as an older version.
Recommendation:
- Handle comparison < 0 separately: warn that local is newer and skip self-update unless --force is explicitly requested.

6. MEDIUM - package.json postinstall diverges from scripts/postinstall.ts
Files:
- package.json:12
- scripts/postinstall.ts:31-38
Evidence:
- scripts/postinstall.ts creates ~/.bun/bin when missing.
- The inline node -e postinstall writes opencode-jce.cmd without ensuring the directory exists.
Impact:
- Windows postinstall can fail in environments where Bun's bin directory has not been created yet.
Recommendation:
- Replace the inline node -e script with bun run scripts/postinstall.ts or mirror its mkdirSync(..., { recursive: true }) behavior.

7. HIGH - OpenCode-native MCP config uses bare binaries that are not installed
Files:
- src/lib/opencode-json-template.ts:137-163
- scripts/merge-config.ts:398-430
- install.ps1:621-633, 689-712
- install.sh:372-391, 439-459
- config/mcp.json:11-47
Evidence:
- opencode.json template uses commands like mcp-server-sequential-thinking, playwright-mcp, mcp-server-github, and mcp-server-memory.
- Installers only run npm cache add, which downloads packages but does not create PATH-accessible binaries.
- config/mcp.json uses npx -y for the same servers, but the actual OpenCode-native opencode.json does not.
Impact:
- Fresh installs can generate opencode.json entries that cannot start MCP servers unless users installed those binaries globally themselves.
Recommendation:
- Use npx -y package commands in opencode.json entries, or change installers to install MCP packages globally. Prefer npx -y to match config/mcp.json and avoid global pollution.

8. HIGH - Existing opencode.json only gets context-keeper, not full default MCP config
Files:
- scripts/merge-config.ts:376-383
- install.ps1:642-662
- install.sh:398-421
- src/commands/update.ts:458-464, 494-506
Evidence:
- ensureOpenCodeJson() returns early if opencode.json already exists.
- Installer registration only adds mcp.context-keeper to existing opencode.json.
- Other default MCP servers are merged into legacy mcp.json, but project notes say OpenCode reads opencode.json key "mcp".
Impact:
- Users with an existing opencode.json do not receive context7, sequential-thinking, playwright, github-search, memory, filesystem, web-fetch, or postgres in the actual OpenCode config.
Recommendation:
- Merge missing default OpenCode-native MCP entries into config.mcp without overwriting existing entries. Use one shared MCP default builder across installer/update paths.

9. MEDIUM - context-keeper may write .opencode-context.md to the wrong directory
Files:
- src/mcp/context-keeper.ts:29-35
- src/lib/opencode-json-template.ts:160-163
- config/mcp.json:3-8
Evidence:
- getProjectRoot() uses PROJECT_ROOT or process.cwd().
- Legacy config/mcp.json sets PROJECT_ROOT.
- OpenCode-native opencode.json template does not set PROJECT_ROOT for context-keeper.
Impact:
- If OpenCode starts the MCP server with cwd set to the config directory or another non-project directory, context_read/update/checkpoint will operate on the wrong .opencode-context.md.
Recommendation:
- Add explicit project root env support to the OpenCode-native context-keeper entry if OpenCode supports interpolation. Otherwise require project root as a tool parameter or detect OpenCode workspace env vars before falling back to cwd.

10. MEDIUM - context_read does not perform documented session-start auto-archive
Files:
- config/AGENTS.md:65-74
- config/skills/context-preservation.md:40-48
- src/mcp/context-keeper.ts:209-227
- src/mcp/context-keeper.ts:364-413
Evidence:
- Instructions require pruning at session start and auto-archiving if the file remains over 50 lines.
- context_read only prunes completed items and warns when over target.
- Archive logic exists only in context_checkpoint.
Impact:
- The enforced MCP behavior does not match documented workflow; oversized context can stay oversized until checkpoint.
Recommendation:
- Extract archive logic to a shared helper and call it from both context_read and context_checkpoint.

11. MEDIUM - Shell injection risk in LSP command detection
Files:
- src/lib/opencode-json-template.ts:58-62
- scripts/merge-config.ts:282-287
- install.sh:682-690
Evidence:
- Code builds shell strings from command values: execSync(`${checkCmd} ${cmd}`) and execSync('command -v ' + entry.command).
- entry.command comes from lsp.json and can be modified.
Impact:
- A malicious or corrupted lsp.json command can execute shell syntax during install/config generation.
Recommendation:
- Use execFileSync/check command with argv instead of string interpolation. Validate LSP command names against a conservative allowlist or schema.

12. MEDIUM - Shell installer embeds unescaped paths into JavaScript snippets
File:
- install.sh:374-391
- install.sh:405-415
Evidence:
- Shell variables such as cli_dir, opencode_json, and context_keeper_path are interpolated directly into JavaScript string literals passed to bun -e.
Impact:
- Paths containing apostrophes, backslashes, or JS-sensitive characters can break install registration. If path components are attacker-controlled, this can become code injection.
Recommendation:
- Pass values through environment variables into the bun -e script instead of interpolating them into JavaScript source.

13. HIGH - Plugin install can delete outside the plugins directory
File:
- src/lib/plugins.ts:108-126
- src/lib/plugins.ts:210-213
Evidence:
- extractRepoName() accepts [\w.-]+, including names like dot or dot-dot path segments.
- repoName is used directly in join(pluginsDir, repoName).
- Existing pluginDir is removed before clone.
Impact:
- A crafted GitHub-looking URL can resolve pluginDir outside pluginsDir and trigger recursive deletion of a parent/sibling directory if it exists.
Recommendation:
- Parse with new URL(), require https://github.com/{owner}/{repo}, reject . and .., reject path separators, and verify resolve(pluginDir).startsWith(resolve(pluginsDir) + sep) before removal or clone.

14. HIGH - Fallback health checks can leak API keys to arbitrary URLs
Files:
- src/lib/fallback.ts:35-40
- src/lib/fallback.ts:107-112
- schemas/fallback.schema.json:30-34
Evidence:
- User fallback.json is parsed without schema validation.
- checkProviderHealth() fetches provider.healthEndpoint and sends Authorization: Bearer ${process.env[provider.apiKeyEnv]}.
- Schema allows arbitrary http/https URLs, and runtime does not enforce even that schema.
Impact:
- A malicious fallback.json can exfiltrate ANTHROPIC_API_KEY, OPENAI_API_KEY, or any environment variable named by apiKeyEnv to an attacker-controlled endpoint.
Recommendation:
- Validate fallback config at load time. Require HTTPS. Prefer fixed provider endpoint mapping or a trusted allowlist. Never send Authorization to arbitrary custom URLs.

15. MEDIUM - Invalid fallback credentials are reported as healthy
File:
- src/lib/fallback.ts:117-125
Evidence:
- Any response with status < 500 is treated as healthy, including 401 and 403.
Impact:
- fallback status and getBestProvider() can select a provider with invalid credentials.
Recommendation:
- Treat 401/403 as unhealthy. Treat 429 as reachable but degraded/rate-limited. Keep 2xx as healthy.

16. MEDIUM - Prompt template traversal check is prefix-based and bypassable
File:
- src/lib/prompts.ts:33-47
Evidence:
- resolvedPath.startsWith(resolvedDir) is used as the boundary check.
- A sibling directory whose name starts with the prompts directory name can pass a string-prefix check.
Impact:
- A crafted template name can read .txt files outside the prompts directory if a sibling path shares the prefix.
Recommendation:
- Use path.relative(resolvedDir, resolvedPath) and reject values starting with .. or absolute paths. Also restrict template names to a simple basename pattern.

17. MEDIUM - Plugin registry and removal use inconsistent identifiers
File:
- src/lib/plugins.ts:108-117
- src/lib/plugins.ts:162-190
Evidence:
- Install directory is based on repoName.
- Registry name is manifest.name.
- Duplicate detection checks p.name === repoName before reading manifest.
- removePlugin(name) deletes join(getPluginsDir(), name), not the original repoName install directory.
Impact:
- If manifest.name differs from repoName, removal can delete the registry entry but leave the cloned directory behind. Duplicate manifest names can also be installed from different repos.
Recommendation:
- Store both manifest.name and installDir/repoSlug. Check duplicates after reading manifest. Remove using the stored install directory.

18. MEDIUM - dashboard and optimize crash when OpenCode DB is missing
Files:
- src/commands/dashboard.ts:129-130
- src/commands/optimize.ts:17-18
- src/lib/tokens.ts:110-119
Evidence:
- new TokenTracker(configDir) auto-detects the OpenCode DB and throws if it is missing.
- dashboard and optimize do not catch this error.
- tokens command handles this case explicitly.
Impact:
- User-facing commands can terminate with an uncaught exception instead of an actionable CLI error.
Recommendation:
- Use the same detect/catch pattern as tokens command and exit with a clear message and EXIT_ERROR.

22. LOW - Semver comparison does not handle prerelease/build metadata
File:
- src/lib/version.ts:155-167
Evidence:
- compareVersions() uses split(".").map(Number).
- Versions like 1.8.9-beta.1 produce NaN segments.
Impact:
- Future prerelease versions can compare incorrectly and skip or misreport updates/migrations.
Recommendation:
- Use a semver parser/comparator or explicitly normalize prerelease/build metadata.

Recommended fix order:
1. Fix self-update/install rollback safety and platform-specific shim handling.
2. Fix OpenCode-native MCP generation to use working npx commands and merge defaults into existing opencode.json.
3. Fix security-sensitive validation: plugin URL/path containment, fallback endpoint allowlist, LSP command execution, install.sh JS interpolation.
4. Fix context-keeper project root and session-start archive mismatch.
5. Fix user-facing CLI behavior for dashboard/optimize/fallback/memory.
6. Add tests for update/install config generation, plugin URL validation, fallback validation, and prompt path containment.

Testing gaps observed:
- No tests cover Unix self-update shim behavior.
- No tests cover interrupted/failed update rollback.
- No tests assert opencode.json MCP commands can actually start after install.
- No tests cover existing opencode.json merging defaults.
- No tests cover malicious plugin URLs, fallback endpoint validation, or prompt traversal.
- No tests cover dashboard/optimize behavior when the OpenCode DB is missing.
