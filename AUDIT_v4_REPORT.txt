OpenCode JCE - Audit v4 Report
Date: 2026-05-04
Version audited: 1.8.11 (commit 8efdab0 on main, synced with origin)
Scope: full re-audit of install/update/version, config/MCP/context, CLI runtime/security/tests/docs.

Verification run:
- bun run typecheck: passed.
- bun test: 163 pass, 0 fail, 1216 expect() calls, 8 files.
- CI validate-json (local): all 23 JSON files PASS.
- bash -n install.sh && bash -n scripts/bundle-offline.sh: passed.
- PowerShell parser check install.ps1: OK.

Summary:
- The critical Anthropic URL typo persists (api.anxthxropic.com) - this was reported in v3 audit but the fix used the same misspelled domain.
- install.sh C# LSP uses wrong binary name (OmniSharp instead of csharp-ls).
- Team sync has path traversal, schema validation, and protocol security gaps.
- Optimizer cost estimation is broken (exact model key matching always returns 0).
- Several medium issues around update workflow, prompt template handling, and config consistency.

Findings (deduplicated across 3 parallel audits, ordered by severity):

1. CRITICAL - Anthropic health endpoint domain is misspelled (non-existent domain)
Files:
- src/lib/fallback.ts:27
- src/lib/fallback.ts:108
- config/fallback.json:6
Evidence:
- Domain is api.anxthxropic.com (with extra x characters: an-x-th-x-ropic) instead of api.anthropic.com.
- TRUSTED_PROVIDER_ENDPOINTS allowlist uses this same misspelled domain.
- Even if user corrects their fallback.json to the real domain, validation rejects it as untrusted.
Impact:
- Anthropic health checks always fail (DNS resolution error).
- Fallback system never selects Anthropic as healthy provider.
- If the misspelled domain is registered by a malicious actor, x-api-key header is sent to it.
Suggested fix:
- Replace api.anxthxropic.com with api.anthropic.com in all three locations.

2. HIGH - install.sh C# LSP uses wrong binary name and install command
File:
- install.sh:488 (LSP_CMDS array uses "OmniSharp")
- install.sh:500 (LSP_INSTALL uses "dotnet tool install -g omnisharp")
Evidence:
- config/lsp.json and install.ps1 both correctly use csharp-ls.
- install.sh still references the old/broken OmniSharp.
Impact:
- Linux/macOS installer never detects existing csharp-ls, always shows C# as not installed.
- If user selects C# for install, wrong tool is installed.
- LSP merge step looks for csharp-ls (from lsp.json) and won't find it.
Suggested fix:
- Change install.sh LSP_CMDS entry to "csharp-ls" and LSP_INSTALL to "dotnet tool install -g csharp-ls --version 0.15.0".

3. HIGH - team pull has no path traversal protection for profile filenames
File:
- src/lib/team.ts:262-275
Evidence:
- Profile filenames from cloned team repo are used directly in join(profilesDir, profile).
- A malicious repo could contain ../../malicious.json which resolves outside profilesDir.
Impact:
- Arbitrary JSON file write outside profiles directory during team pull.
Suggested fix:
- Validate resolve(join(profilesDir, profile)).startsWith(resolve(profilesDir) + sep) before writing.

4. HIGH - team branch name regex allows path traversal sequences
File:
- src/lib/team.ts:100, 211
Evidence:
- Regex /^[a-zA-Z0-9._\/-]+$/ allows .. sequences.
- Branch name is passed to git clone --branch and git push origin.
Impact:
- Crafted team.json branch name could cause unexpected git behavior.
Suggested fix:
- Add explicit rejection of branch names containing "..".

5. MEDIUM - optimizer estimateCost always returns 0 (exact model key matching broken)
Files:
- src/lib/optimizer.ts:39-42
- src/lib/constants.ts (COST_PER_1K keys)
Evidence:
- COST_PER_1K has short keys like "claude-sonnet", "gpt-4o".
- Actual model IDs from DB are full identifiers like "claude-sonnet-4-20250514".
- Lookup COST_PER_1K[model] always returns undefined.
Impact:
- optimize command shows 0% savings for all suggestions.
- Cost estimates are meaningless.
Suggested fix:
- Use fuzzy/prefix matching: find first COST_PER_1K key where model.includes(key).

6. MEDIUM - mergeDirectory never returns -1, fetch failures silently swallowed
File:
- src/commands/update.ts:420, 573-584
Evidence:
- mergeDirectory returns 0 when fetchDirectoryListing returns [] (which happens on network error).
- Callers check < 0 to detect failures but mergeDirectory never returns negative.
Impact:
- Offline update for profiles/prompts/skills silently reports success.
Suggested fix:
- Return -1 when fetchDirectoryListing returns [] and local directory is empty/missing.

7. MEDIUM - update backs up entire config dir (including node_modules) on every run
File:
- src/commands/update.ts:714-724
Evidence:
- cp(configDir, backupDir, { recursive: true }) copies everything including cli/node_modules.
- Old backups are never cleaned up.
Impact:
- Each update creates hundreds of MB backup. Accumulates over time.
Suggested fix:
- Exclude cli/ from backup (it has its own staging/rollback). Add backup rotation.

8. MEDIUM - package.json postinstall uses CJS require() but package is "type": "module"
File:
- package.json:4,12
Evidence:
- "type": "module" at line 4.
- postinstall uses node -e with require('fs').
- Node.js >= 22 may enforce ESM in -e context.
Impact:
- Postinstall can fail on newer Node.js versions.
Suggested fix:
- Use "postinstall": "bun run scripts/postinstall.ts" instead of inline node -e.

9. MEDIUM - scripts/postinstall.ts exists but is never used (duplicated logic)
File:
- scripts/postinstall.ts vs package.json:12
Evidence:
- Both implement the same Windows shim logic independently.
Impact:
- Maintenance risk: fixes in one don't propagate to the other.
Suggested fix:
- Wire package.json postinstall to use scripts/postinstall.ts.

10. MEDIUM - team pull does not validate pulled configs against schemas
File:
- src/lib/team.ts:247-251

Impact:
